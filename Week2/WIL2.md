## __1. 컨트롤러, 서비스, 리포지토리__
### __1.1. 컨트롤러__
사용자의 요청을 처리한 후 지정된 뷰에 모델 객체를 넘겨주는 역할을 한다. 사용자의 요청이 진입하는 지점이며 요청에 따라 어떤 처리를 할지 결정하도록 서비스에 넘겨준다. 서비스에서 처리된 내용은 다시 컨트롤러로 넘어가게 된다.<br/><br/>

### __1.2. 서비스__
서비스에는 사용자의 요구사항을 처리하는 로직이 존재한다. 사용자의 요구사항을 처리할 때 데이터베이스의 데이터가 필요한 경우 리포지토리에 요청을 하게 된다.<br/><br/>

### __1.3. 리포지토리__
데이터베이스에 접근하는 메서드를 사용하기 위한 인터페이스이다. 데이터베이스의 연결과 해제, 그리고 자원을 관리하는 역할을 한다.<br/><br/>

## __2. TDD(Test Driven Development)__
테스트 케이스에 맞추어 실제 개발 단계로 이행하는 개발 방법론을 말한다. 테스트 케이스만을 완벽하게 수행하는 것을 목표로 한다. 하나의 테스트를 통과하게 되면 테스트 케이스를 확장하여 새로운 테스트를 진행하는 과정을 반복하여 큰 규모의 프로젝트를 완성하게 된다.

### __2.1. 절차__
1. 테스트 케이스 작성
2. 테스트 케이스를 통과하는 코드 작성
3. 작성한 코드 리팩토링
4. 테스트 케이스를 확장하여 위의 과정 반복<br/><br/>

### __2.2. 장점__
* 코드의 유지보수 용이
* 프로그래밍 시간 단축
* 뛰어난 프로그램 소스코드 기록<br/><br/>

# __:pushpin: WIL__
## __1. 회원 관리 예제__
### __1.1. 비즈니스 요구사항__
* 데이터: 회원, ID, 이름
* 기능: 회원 등록, 조회
* 데이터 저장소가 선정되지 않은 상태 가정, 따라서 인터페이스로 구현 클래스를 변경할 수 있도록 설계<br/><br/>

### __1.2. 도메인 구현__
* 회원 클래스 정의
* ID, 이름 멤버 변수 정의
* Getter and Setter 이용하여 ID와 이름을 반환, 설정하는 함수 정의<br/><br/>

### __1.3. 리포지토리 구현__
* 회원 정보 저장 함수 정의
* 회원 정보를 ID, 이름으로 찾는 함수 정의
* 회원 정보 목록을 반환하는 함수 정의
* 저장한 정보를 지우는 함수 정의(test용)<br/><br/>

### __1.4. 서비스 구현__
* 회원가입 함수 정의(이미 존재하는 회원일 경우 오류 메시지 반환)<br/><br/>

### __1.5. 테스트__
테스트 폴더에 테스트 케이스 파일을 작성하여 테스트를 진행한다. 테스트는 함수 단위로도 가능하고 클래스 단위로도 가능하다. 정상적인 상황뿐만 아니라 예외 상황에 대해서도 테스트를 진행한다.
